DATA FLOW
=========

USER REGISTRATION:
  register.html
      ↓
  [Enter email/password]
      ↓
  js/auth.js
      ↓
  POST /api/auth/register
      ↓
  auth.controller.js → auth.service.js
      ↓
  Hash password + create user
      ↓
  Return JWT token
      ↓
  Save to localStorage
      ↓
  Redirect to games.html


GAME BROWSING:
  games.html
      ↓
  [Page loads]
      ↓
  js/games.js calls api.js
      ↓
  GET /api/games?title=X&platform=Y
      ↓
  games.controller.js → games.service.js
      ↓
  Query SQLite
      ↓
  Return game list
      ↓
  Render cards + filters


RENTING A GAME:
  games.html
      ↓
  [Click Rent button]
      ↓
  js/games.js calls api.js
      ↓
  POST /api/rentals { gameId: 1 }
      ↓
  authMiddleware ✓ verifies JWT
      ↓
  rentals.controller.js → rentals.service.js
      ↓
  Validate: game exists, stock > 0, no active rental
      ↓
  Decrement game.stock_available
      ↓
  Create rental (due_at = now + 7 days)
      ↓
  Return rental object
      ↓
  Update games.html + show success


ADMIN MANAGING GAMES:
  admin.html
      ↓
  [Check auth = ADMIN ✓]
      ↓
  [Click Add Game]
      ↓
  Fill form + submit
      ↓
  POST /api/games
      ↓
  authMiddleware ✓
      ↓
  adminOnly middleware ✓
      ↓
  games.controller.js → games.service.js
      ↓
  Create/update/delete in SQLite
      ↓
  Return updated game
      ↓
  Refresh games list

AUTHENTICATION FLOW
===================

1. REGISTER/LOGIN:
   ┌─────────────────────────────────────────┐
   │ User enters email + password            │
   │         ↓                                │
   │ POST /api/auth/register or /login       │
   │         ↓                                │
   │ Backend hashes password (bcryptjs)      │
   │         ↓                                │
   │ Check credentials                       │
   │         ↓                                │
   │ Generate JWT token (7-day expiry)       │
   │         ↓                                │
   │ Return { token, user }                  │
   │         ↓                                │
   │ Frontend saves token to localStorage    │
   └─────────────────────────────────────────┘

2. PROTECTED REQUESTS:
   ┌─────────────────────────────────────────┐
   │ Frontend includes token in header       │
   │   Authorization: Bearer <token>         │
   │         ↓                                │
   │ authMiddleware receives request         │
   │         ↓                                │
   │ Extract and verify JWT                  │
   │         ↓                                │
   │ Signature valid? ✓                      │
   │         ↓                                │
   │ Decode and attach user to req.user      │
   │         ↓                                │
   │ Continue to controller                  │
   └─────────────────────────────────────────┘

3. ADMIN-ONLY REQUESTS:
   ┌─────────────────────────────────────────┐
   │ authMiddleware passes ✓                 │
   │         ↓                                │
   │ adminOnly middleware checks role        │
   │         ↓                                │
   │ Is role === 'ADMIN'? ✓                  │
   │         ↓                                │
   │ Yes → continue to handler               │
   │ No → return 403 Forbidden               │
   └─────────────────────────────────────────┘

For Admin Routes:
  Request → authMiddleware ─→ Verified ✓ ─→ adminOnly ─→ Yes ✓ → Controller
                ↓                              ↓

STOCK MANAGEMENT
================

Initial State:
  Game: Elden Ring
  stock_total: 5
  stock_available: 5
  ✓ Can rent

After Rent (stock_available > 0):
  POST /api/rentals { gameId: 1 }
       ↓
  Check: stock_available (5) > 0? YES ✓
       ↓
  Decrement: stock_available = 5 - 1 = 4
       ↓
  Create rental
       ↓
  Update games table: stock_available = 4

After Return:
  POST /api/rentals/:id/return
       ↓
  Increment: stock_available = 4 + 1 = 5
       ↓
  Update rental: status = 'RETURNED', returned_at = now
       ↓
  games table updated: stock_available = 5


OVERDUE DETECTION
=================

Rental Created:
  rented_at:  2026-01-14 10:00:00 UTC
  due_at:     2026-01-21 10:00:00 UTC (rented_at + 7 days)
  returned_at: NULL
  
Check Overdue:
  Current date > due_at AND returned_at IS NULL?
  
  2026-01-20: NOT overdue (before due date)
  2026-01-21: NOT overdue (on due date)
  2026-01-22: ✓ OVERDUE (after due date, not returned)
  
  After return: returned_at = 2026-01-22 15:00:00
  Check: NOT overdue (returned_at is set)

Display:
  ⚠️ OVERDUE label in red
  Card highlighted in red background
